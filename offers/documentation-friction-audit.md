# Documentation Friction Audit

## Purpose

Identify where documentation forces users to infer system behavior.

This is a short diagnostic offer for teams with complex product, API, or workflow documentation.

## Best fit

This offer is useful when a team sees:

- repeated support questions
- unclear onboarding
- fragmented product documentation
- inconsistent explanations across pages
- API behavior that is hard to anticipate
- documentation that is correct locally but confusing globally

## Scope

Audit a focused documentation area:

- one product area
- one API workflow
- one onboarding path
- one permission model
- one lifecycle or state model

## Method

1. Review selected documentation pages
2. Identify high-friction points
3. Classify friction types
4. Map implicit relationships
5. Prioritize issues by impact and effort
6. Recommend structural corrections

## Deliverables

- 10 documented friction points
- friction classification
- impact/effort prioritization
- 1–3 behavior models or matrices
- suggested structural corrections
- short executive summary

## Duration

Typical duration:

```text
5 working days
```

## Example output

For a payment API:

```text
role + amount + approval rule → status → webhook → next action
```

## Not included

- full documentation rewrite
- implementation in the client CMS
- product specification rewriting
- support process redesign

## Value

The audit produces a map of where users lose predictability.

It gives the team a prioritized path before rewriting documentation.
